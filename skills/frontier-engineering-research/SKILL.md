# Frontier Engineering Research

## Overview

Find frontier engineering ideas without drowning in generic search results. Use X to discover live signals, then verify practical claims against official docs, GitHub, release notes, and real engineering writeups.

## Research Order

1. Start with X for fresh signals: core maintainers, respected practitioners, company engineers, tool authors, and high-signal threads.
2. Promote only claims that repeat across credible people or connect to concrete artifacts.
3. Verify implementability with official docs, GitHub activity, releases, issues/discussions, benchmarks, or production engineering posts.
4. Use Chinese sources only for local context or translation checks, not as primary evidence.
5. Skip papers and academic conference material unless the user explicitly asks or it is the origin of a production-ready technique.

## Source Quality

Prefer:

- X posts from builders close to the implementation or adoption decision.
- GitHub repositories with recent commits, active issues, serious discussions, and real users.
- Official docs, changelogs, migration guides, and release notes.
- Engineering blogs from teams that operate the solution in production.
- HN/Reddit/Lobsters only when X does not expose enough debate or failure cases.

Reject or downgrade:

- SEO comparison pages, vendor fluff, content farms, and thin listicles.
- Unverified viral takes with no code, docs, adoption, or reproducible experience.
- Benchmarks without workload details.
- Outdated posts when the ecosystem has moved.

## Output

Keep the result concise:

- Best sources to follow: 5-10 links or accounts, grouped by why they matter.
- Market map: 3-5 engineering options, including the incumbent/default path.
- Recommendation: one best current direction and when not to choose it.
- Evidence: short notes tying claims back to X signals plus implementation proof.
- Open risks: unknowns worth testing before committing.

If the user asks only for sources, stop at the source list and explain what each source is good for.

## Guardrails

- Treat X as discovery, not proof.
- State source freshness when recency affects the answer.
- Call out hype explicitly when adoption or maintainability is unclear.
- Prefer practical engineering tradeoffs over abstract trend summaries.
